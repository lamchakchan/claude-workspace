# Rules

Claude Code natively supports `.claude/rules/` as a directory for modular instruction files. Any `.md` file placed in this directory is automatically loaded into Claude's context at session start. This document explains how the platform uses rules, how to add your own, and how path-scoped rules work.

## 1. What `.claude/rules/` Is

`.claude/rules/` is a **native Claude Code feature**. Any Markdown file placed in this directory is discovered automatically and loaded into Claude's context. No extra configuration is needed -- Claude Code reads them at session start just like it reads `CLAUDE.md` files.

Key properties:

- Files are discovered recursively (subdirectories are supported)
- Each file should cover **one topic** (e.g., API conventions, test patterns, security rules)
- Files are committed to git and shared with the team
- Rules are **additive** to CLAUDE.md -- they do not replace it

```
.claude/rules/
  platform.md          # Platform conventions (auto-generated by attach)
  api-conventions.md   # Your team's API style guide
  test-patterns.md     # Testing conventions
  security/
    auth-rules.md      # Auth-specific rules (path-scoped)
    input-validation.md
```

## 2. How the Platform Uses Rules

The `claude-workspace attach` command generates a `platform.md` rules file containing platform conventions (team execution, plan conventions, MCP tool preferences).

The behavior depends on whether your project already has a `.claude/CLAUDE.md`:

| Scenario | What `attach` writes | Why |
|----------|---------------------|-----|
| **No `.claude/CLAUDE.md` exists** (first-time setup) | Full scaffold to `.claude/CLAUDE.md` | Fresh project -- one file with everything |
| **`.claude/CLAUDE.md` already exists** | Platform conventions to `.claude/rules/platform.md` | Non-destructive -- preserves your existing instructions |
| **`--force` flag** | Full scaffold to `.claude/CLAUDE.md` | Explicit overwrite requested |

This means re-running `attach` on an established project will **never clobber your existing CLAUDE.md**. Platform conventions are delivered as a separate rules file instead.

The `enrich` command follows the same logic: if `.claude/CLAUDE.md` exists, enrichment output goes to `.claude/rules/platform.md`.

## 3. Adding Custom Rules

Add your own rules by creating `.md` files in `.claude/rules/`. Each file covers one topic.

**Example: API conventions**

Create `.claude/rules/api-conventions.md`:

```markdown
# API Conventions

- All endpoints must return JSON with a top-level `data` or `error` key
- Use HTTP status codes correctly (201 for creation, 404 for not found, 422 for validation errors)
- All list endpoints must support pagination with `limit` and `offset` query parameters
- Rate limiting headers must be included in every response
```

**Example: Test patterns**

Create `.claude/rules/test-patterns.md`:

```markdown
# Test Patterns

- Unit tests are co-located with source files (e.g., `user.ts` and `user.test.ts`)
- Integration tests live in `tests/integration/`
- Use factory functions for test data, not raw object literals
- Mock external services at the HTTP layer, not the function layer
```

Commit these files to git so the whole team benefits.

## 4. Path-Scoped Rules

Rules can be scoped to specific files using YAML frontmatter with a `paths:` array. These rules are **only loaded when Claude reads or edits files matching the glob patterns**. This keeps context lean -- Claude only sees rules relevant to the files being worked on.

### Syntax

```markdown
---
paths:
  - "glob/pattern/**/*.ext"
---

# Your rules here
```

### Examples

**Frontend rules** (`.claude/rules/frontend.md`):

```markdown
---
paths:
  - "src/components/**/*.tsx"
  - "src/components/**/*.css"
---

# Frontend Rules

- Use functional components with hooks, never class components
- CSS modules for styling, no inline styles
- Every component must have a corresponding Storybook story
- Use `React.memo()` only when profiling shows a measurable benefit
```

**Security rules for auth code** (`.claude/rules/security/auth-rules.md`):

```markdown
---
paths:
  - "src/auth/**/*"
  - "src/middleware/auth*"
---

# Auth Security Rules

- Never store plaintext passwords -- always use bcrypt with cost >= 12
- Session tokens must be cryptographically random (use `crypto.randomBytes`)
- All auth endpoints must be rate-limited
- JWT secrets must come from environment variables, never hardcoded
- Log authentication failures but never log credentials or tokens
```

**Test conventions** (`.claude/rules/test-conventions.md`):

```markdown
---
paths:
  - "**/*_test.go"
  - "**/*.test.ts"
  - "tests/**/*"
---

# Test Conventions

- Table-driven tests with descriptive case names
- No network calls in unit tests -- mock at the boundary
- Test the public API, not internal implementation details
- Each test must clean up any resources it creates
```

**Database migration rules** (`.claude/rules/migrations.md`):

```markdown
---
paths:
  - "migrations/**/*"
  - "prisma/**/*"
  - "db/**/*"
---

# Migration Rules

- Migrations must be reversible (include both up and down)
- Never modify existing migrations -- always create new ones
- Add indexes concurrently in production (CREATE INDEX CONCURRENTLY)
- Test migrations against a copy of production data before deploying
```

**Infrastructure rules** (`.claude/rules/infra.md`):

```markdown
---
paths:
  - "Dockerfile*"
  - "docker-compose*.yml"
  - ".github/workflows/**/*"
  - "terraform/**/*"
---

# Infrastructure Rules

- Pin all base image versions (no `latest` tags)
- Multi-stage builds for production Dockerfiles
- CI workflows must have timeout limits
- Terraform resources must have tags for cost attribution
```

### How path scoping works

- Glob patterns follow standard matching rules (`*` matches within a directory, `**` matches across directories)
- Multiple patterns in the `paths:` array are OR-ed -- matching any one pattern loads the rule
- Path-scoped rules are loaded **in addition to** unconditional rules, not instead of them
- Subdirectories work: `.claude/rules/security/auth-rules.md` with a `paths:` frontmatter is valid

## 5. Unconditional Rules

Rules **without** YAML frontmatter are loaded at session start unconditionally, the same way CLAUDE.md files are. Use unconditional rules for conventions that apply everywhere in the project.

```markdown
# Code Style

- Use 2-space indentation
- Prefer early returns over deeply nested conditionals
- Error messages must be lowercase and not end with punctuation
```

No frontmatter means this file loads every session, regardless of which files Claude is working on.

## 6. User-Level Rules

Claude Code also supports user-level rules at `~/.claude/rules/`. These apply to **all projects** on the machine, similar to how `~/.claude/CLAUDE.md` applies globally.

Use user-level rules for personal preferences that apply everywhere:

```
~/.claude/rules/
  style-preferences.md     # "I prefer concise responses"
  tool-preferences.md      # "Always use bun instead of npm"
```

User-level rules are never committed to any repository. They are personal to you.

## 7. Relationship to CLAUDE.md

Rules are **additive**, not a replacement for CLAUDE.md. Here is how they relate:

| File | Purpose | When to use |
|------|---------|-------------|
| `CLAUDE.md` (project root) | Platform-level context | Managed by the platform (root-level instructions) |
| `.claude/CLAUDE.md` | Core project instructions | Tech stack, build commands, conventions, key directories |
| `.claude/rules/*.md` | Modular topic rules | Topic-specific conventions, path-scoped rules, platform conventions |
| `~/.claude/CLAUDE.md` | Global personal instructions | Cross-project preferences |
| `~/.claude/rules/*.md` | Global personal rules | Cross-project topic rules |

**CLAUDE.md** is for core project instructions that every session needs: tech stack, build commands, directory layout, and primary conventions.

**Rules** are for modular, topic-specific instructions that can be organized, scoped, and maintained independently. They are especially valuable for:

- **Path-scoped conventions** -- rules that only apply when working on specific files
- **Team-contributed guidelines** -- each team member can add rules for their domain
- **Platform conventions** -- delivered by `attach` without touching your CLAUDE.md

The full load order (all concatenated, not overriding):

```
~/.claude/CLAUDE.md           # Global user instructions
~/.claude/rules/*.md          # Global user rules
CLAUDE.md                     # Project root
.claude/CLAUDE.md             # Project config
.claude/rules/*.md            # Project rules (unconditional ones at session start,
                              #   path-scoped ones when matching files are accessed)
```

## 8. Migration Guide

### For users who already ran `attach` before this change

If you ran `attach` on a project before rules support was added:

1. **Re-run `attach`** on your project:
   ```bash
   claude-workspace attach /path/to/your/project
   ```

2. Since your project already has a `.claude/CLAUDE.md`, the command will create `.claude/rules/platform.md` with platform conventions while **preserving your existing CLAUDE.md untouched**.

3. If you want a completely fresh scaffold (replacing your CLAUDE.md), use `--force`:
   ```bash
   claude-workspace attach /path/to/your/project --force
   ```

### For users migrating from CLAUDE.local.md

`CLAUDE.local.md` is deprecated. Claude Code still loads it if present (for backward compatibility), but the platform no longer creates new ones.

**Recommended alternatives:**

| What you used CLAUDE.local.md for | Use instead |
|-----------------------------------|-------------|
| Personal project overrides (local ports, skip tests) | `.claude/rules/` with a gitignored file, or continue using `CLAUDE.local.md` (still loaded) |
| Team-wide conventions | `.claude/rules/*.md` (committed to git) |
| File-specific rules | `.claude/rules/*.md` with `paths:` frontmatter |

### For users adding rules to existing projects

Just create `.claude/rules/` and start adding `.md` files. No `attach` required. Claude Code discovers them automatically.

```bash
mkdir -p .claude/rules
echo '# My Rules' > .claude/rules/my-conventions.md
```
